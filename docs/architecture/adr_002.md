# ADR-002: Algorytm Słowa Dnia oparty na Dacie

**Status:** ✅ Zaimplementowane  
**Data:** 2026-04-13  
**Decyzja:** Wybór słowa na podstawie wyliczenia dni od daty startowej (`daysElapsed`) zamiast losowania, backendowego harmonogramowania, czy przypisywania dat do każdego słowa

## Kontekst

Każdego dnia użytkownik powinien zobaczyć **to samo słowo**, niezależnie od tego, kiedy w ciągu dnia wejdzie do aplikacji. Słowa są posortowane w tablicy (1-100) i reprezentują kolejną sekwencję. Brak backendu do harmonogramowania, a aplikacja musi działać offline.

## Decyzja

1. **Data startowa:** `START_DATE = new Date('2026-04-13')`
2. **Indeks słowa:** `daysElapsed = floor((now - START_DATE) / 86400000)`
3. **Selekcja:** `word = words[daysElapsed]` (z obsługą edge cases)
4. **Edge case - przed startem:** `daysElapsed < 0` → `words[0]` (pierwsze słowo)
5. **Edge case - po wyczerpaniu:** `daysElapsed >= words.length` → `null` (tryb archiwum)

Implementacja w `useWordOfDay.js`:
```javascript
const getDaysElapsed = (startDate) => {
  const now = new Date();
  now.setHours(0, 0, 0, 0);
  const start = new Date(startDate);
  start.setHours(0, 0, 0, 0);
  return Math.floor((now - start) / (1000 * 60 * 60 * 24));
};
```

## Konsekwencje

### Pozytywne
- **Spójność:** Wszyscy użytkownicy widzą to samo słowo w danym dniu (kluczowe dla "słowa dnia")
- **Offline-first:** Zero zależności od czasu serwera czy synchronizacji
- **Deterministyczne:** Łatwe debugowanie i przewidywanie (np. "1 maja to słowo X")
- **Brak harmonogramowania:** Nie potrzebny cron job, queue, ani scheduler
- **Proste cachowanie:** Browser cache działa naturalnie, API nie jest potrzebne
- **Testowalność:** Łatwe symulowanie różnych dat w testach (wystarczy zmienić `now`)

### Negatywne
- **Zależność od daty lokalnej:** Użytkownicy w różnych strefach czasowych mogą (teoretycznie) widzieć różne słowa, jeśli wejdą tuż po północy ich czasu lokalnego. W praktyce akceptowalne – każdy człowiek ma "swoje" słowo dnia względem swojego dnia.
- **Twarda kolejność:** Słowa zawsze w tej samej kolejności. Brak możliwości "przypadkowego dnia" bez zmiany algorytmu.
- **Aktualizacja bazy:** Jeśli dodamy więcej słów w połowie roku, daty startowe muszą zostać zachowane (słowa 1-100 mają już swoje dni). Nowe słowa (101+) będą kontynuacją.
- **Brak możliwości podwójnego słowa:** Jeśli użytkownik przegapi dzień, słowo przepadło – algorytm nie "zapamiętuje" tego.

## Alternatywy Rozważone

### 1. Losowe słowo codziennie (z seedem opartym na dacie)
- **Plusy:** Każdego dnia inne słowo, nadal deterministyczne dla danej daty
- **Minusy:** Trudniej o przewidywalność (np. "jaki dzisiaj temat słów?"), słowa mogą się powtarzać przed 100 dniami
- **Dlaczego odrzucono:** Chcemy kontrolować sekwencję – słowa idą od najłatwiejszych/najpopularniejszych do najtrudniejszych/niszowych. Losowe psułoby narrację.

### 2. Backendowy harmonogram (cron job wybierający słowo)
- **Plusy:** Pełna kontrola z poziomu serwera, łatwe aktualizacje, jeden wybór dla wszystkich
- **Minusy:** Wymaga serwera 24/7, potencjał błędów przy synchronizacji czasów (timezones), brak offline
- **Dlaczego odrzucono:** Nie ma backendu w MVP. Overkill dla prostego wyboru słowa.

### 3. Przypisanie daty do każdego słowa w bazie
- **Plusy:** Elastyczność (można przesuwać słowa), łatwe aktualizacje (zmiana przypisania)
- **Minusy:** Trzeba przechowywać daty, logikę przesunięć, potencjalne luki (jeśli brak słowa na dany dzień)
- **Dlaczego odrzucono:** Zbyt złożone dla 100 słów. Daty są w rzeczywistości sekwencyjne, więc matematyczny wzór wystarcza.

### 4. Algorytm oparty na hashu daty (np. `hash('2026-04-28') % 100`)
- **Plusy:** Pseudolosowość, determinizm, brak konieczności znajomości kolejności
- **Minusy:** Słowa mogłyby się powtarzać w ciągu 100 dni, trudniejsze zapewnienie równomiernego rozkładu
- **Dlaczego odrzucono:** Brak kontroli nad tym, czy słowo już się pojawiło, utrata narracji "od najpopularniejszych do najrzadszych".

## Planowane Rozwiązanie w Fazy 2

Jeśli aplikacja się udowodni i konieczna będzie większa elastyczność:

1. **Słowa tematyczne:** Możliwość zmiany motywu/przebiegu słów (np. "tydzień czasowników", "tydzień archaizmów") – wymagałoby przypisania słów do grup/tytułów.
2. **Podwójne słowo (alternatywa):** Algorytm mógłby wybierać z puli 2-3 słów na dany dzień, aby użytkownicy przegapili jedno, mogli zobaczyć drugie.
3. **Personalizacja:** Na podstawie tego, które słowa użytkownik już zna (z ulubionych/albo quizu) – ale wymaga backendu i profilu.
4. **Harmonogram w CMS:** Zamiast obliczania `daysElapsed`, backend/CMS przypisywałby słowo do konkretnej daty (pozwalałoby na przesuwanie terminów, pomijanie dni, itp.).

Migracja wymagałaby dodania endpointu `/api/word-of-day?date=YYYY-MM-DD` i zmiany `useWordOfDay` z obliczania lokalnego na fetch z fallbackem do lokalnego (zachowując offline-first). Implementacja: ~1 dzień pracy.
