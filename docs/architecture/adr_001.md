# ADR-001: Brak Backendu w MVP

**Status:** ✅ Zaimplementowane  
**Data:** 2026-04-13  
**Decyzja:** Zastosowanie architektury bez serwera (client-only) z lokalnymi danymi i localStorage zamiast backendu z bazą danych i API

## Kontekst

MVP ma za zadanie zweryfikować koncept "Słowa Dnia" – aplikacji pomagającej w nauce pięknych polskich słów. Ograniczenia:
- Czas budowy: krótki (dni/tygodnie, nie miesiące)
- Budżet: minimalny (chcemy uniknąć kosztów serwera)
- Zespół: mały (brak zasobów do utrzymania backendu i DevOps)
- Słowa: statyczne, zmieniane codziennie według deterministycznego algorytmu
- Ulubienia: proste zapytania CRUD dla użytkownika

## Decyzja

1. **Baza słów** jako `src/data/words.json` importowana do bundla (Vite)
2. **Stan użytkownika** (ulubione słowa) w `localStorage` przeglądarki
3. **Zero backendu** – brak API, bazy danych, ani serwera aplikacyjnego
4. **Hosting** na Vercel (statyczne assety, CDN, HTTPS automatycznie)
5. **Routing** klientowy (React Router) – brak SSR

## Konsekwencje

### Pozytywne
- **Mniejszy koszt**: Vercel darmowy dla statycznych stron, brak serwera (0$ miesięcznie vs 5-50$ dla VPS/DB)
- **Szybszy czas wdrożenia**: Brak potrzeby projektowania API, schematów bazy, endpointów, autoryzacji
- **Offline-first**: Aplikacja działa w 100% offline (dane w bundlu, stan w localStorage)
- **Brak problemów z skalowaniem**: SPA na CDN skaluje się do milionów użytkowników bez dodatkowej pracy
- **Brak wektorów ataku serwerowych**: Zero API = zero podatności na SQL Injection, XSS po stronie serwera, DDoS na endpointy
- **Proste deployment**: Git push → Vercel build → live (CI/CD wbudowane)

### Negatywne
- **Brak synchronizacji multi-device**: Ulubione na desktopie nie przenoszą się na mobile (localStorage przeglądarki-lokalny)
- **Aktualizacja słów wymaga redeployu**: Zmiana bazy `words.json` lub logiki wymaga nowego buildu i deployu (vs API gdzie wystarczy edycja w CMS)
- **Limit localStorage**: ~5MB – wystarczające na 100 słów (50KB), ale problematyczne gdybyśmy chcieli przechowywać tysiące customowych wpisów
- **Brak analityki po stronie serwera**: Nie wiemy, co robią użytkownicy, brak śledzenia eventów w czasie rzeczywistym
- **SEO ograniczone**: SSR brak, choć dla aplikacji typu "wyszukiwarka słów" to akceptowalne

## Alternatywy Rozważone

### 1. Firebase / Firestore (Backend jako usługa)
- **Plusy**: Autentykacja, realtime DB, hosting, darmowy tier generozny
- **Minusy**: Koszt przy skali (Firestore czyta/zapisy płatne), vendor lock-in, wymagałby refaktoryzacji przy ew. migracji
- **Dlaczego odrzucono**: Overkill dla 100 słów i prostych ulubionych. MVP chce zweryfikować koncept, nie skalowalność.

### 2. Node.js + Express + PostgreSQL (Własny backend)
- **Plusy**: Pełna kontrola, łatwe migracje schematów, własne endpointy
- **Minusy**: Koszty hostingowe (VPS min 5-10$/m), utrzymanie bazy danych (backupy, patche), DevOps (monitoring, logi)
- **Dlaczego odrzucono**: Zbyt wolny time-to-market, zbyt wysoki koszt utrzymania dla MVP.

### 3. Headless CMS (Contentful, Strapi, Sanity) + API
- **Plusy**: Edycja słów przez UI (nie przez JSON), łatwe aktualizacje, wersjonowanie treści
- **Minusy**: Koszty po przekroczeniu darmowego limitu (Contentful: 2000 wierszy w darmowym, potem $$), wymaga fetchu przy starcie
- **Dlaczego odrzucono**: Obecnie 100 słów to stały zbiór – nie potrzebujemy UI dla edytorów. Koszt Contentful przewyższa zaoszczędzoną pracę.

## Planowane Rozwiązanie w Fazy 2 (Gdy MVP się udowodni)

Jeśli aplikacja zdobędzie użytkowników i konieczna będzie synchronizacja multi-device oraz dynamiczna aktualizacja słów:

1. **Backend**: Node.js + Express (lub Firebase)
2. **Baza**: PostgreSQL (lub Firestore)
3. **Autoryzacja**: OAuth (Google) lub email
4. **Synchronizacja**: REST API dla słów, WebSocket dla realtime ulubionych
5. **CMS**: Contentful dla edycji słów przez non-tech team
6. **Migracja**: Adapter pattern – zachowanie API `useFavorites` identycznego, zmiana implementacji z localStorage na API calls

Migracja będzie transparentna dla użytkowników (darmowy tier backendu na start), a dla deweloperów wymaga zmiany implementacji hooków (ok. 2 dni pracy).
